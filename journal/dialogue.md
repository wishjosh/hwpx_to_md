### 빌드 방식: 단일 HTML vs 모듈화 프로젝트

**배경**
기존 `hwpx2md.html`은 단일 파일로 배포·실행이 간편하지만, 기능이 늘어날수록 코드 관리가 어려워지는 구조였다.

**논의**
- 단일 HTML: 배포·실행이 쉽고 오프라인 완전 동작. 단, 코드가 거대해질수록 유지보수 부담이 커짐.
- 모듈화 + 번들러: 소스는 파일별로 깔끔하게 분리하고, 빌드 결과물만 단일 파일로 유지할 수 있음.

**결정**: Vite + `vite-plugin-singlefile` 하이브리드 방식 채택
- 개발 시에는 모듈화된 소스를 편집
- 배포 시에는 `npm run build`로 단일 `dist/index.html` 생성
- 사용자는 기존처럼 파일 하나만 받아서 더블클릭으로 실행

### 프리뷰 UI: 탭 뷰 vs 분할 뷰 vs 둘 다

**배경**
변환된 마크다운을 보여줄 때 소스 코드와 렌더링 결과를 어떻게 배치할 것인가.

**논의**
- 탭 뷰만: 구현이 단순하고 공간을 넓게 쓸 수 있지만, 소스와 프리뷰를 동시에 볼 수 없음.
- 분할 뷰만: 실시간 비교가 가능하지만, 좁은 화면에서 불편.
- 두 가지 모두 지원: 사용자가 상황에 맞게 선택 가능. 구현 복잡도가 약간 높아지지만 UX가 훨씬 좋음.

**결정**: 탭(에디터/프리뷰)과 분할 뷰 모두 지원
- 데스크톱(768px 이상)에서는 변환 완료 시 자동으로 분할 뷰 활성화
- 모바일에서는 분할 뷰 탭을 숨기고 탭 뷰만 노출 (미디어 쿼리)

### 에디터: 읽기 전용 div vs 편집 가능한 textarea

**배경**
변환 결과물을 사용자가 직접 수정할 수 있게 할지 여부.

**논의**
- 읽기 전용 `div`: 복사·다운로드만 제공. 수정이 필요하면 별도 에디터를 써야 함.
- 편집 가능한 `textarea`: 변환 직후 경미한 오류나 원하는 내용을 바로 수정 가능. 수정된 내용이 복사·다운로드에 반영되어야 함.

**결정**: `textarea` 기반 에디터로 변경
- 변환 결과를 즉시 수정·가공할 수 있어 실용성이 높아짐
- `input` 이벤트로 실시간 렌더링 동기화
- 복사·다운로드는 전역 변수가 아닌 에디터의 현재 `.value`를 참조

### 개발일지 vs 논의일지 분리 여부

**배경**
프로젝트 문서화 방식을 정리할 필요가 있었음.

**결정**: 둘 다 생성 (`journal/development.md`, `journal/dialogue.md`)
- `development.md`: 무엇을 만들었는지 (구현 이력)
- `dialogue.md`: 왜 이렇게 만들었는지 (설계 근거 및 대안 비교)
- 다른 프로젝트에서도 동일 규칙을 사용 중이므로 일관성 유지
